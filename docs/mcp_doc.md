# MCP 服务 🔧(Model Context Protocol) 服务使用说明文档

**文档更新时间：2026年3月8日**

**Anywhere** 引入了对 **MCP (Model Context Protocol)** 的深度支持。这是一项革命性的功能，它打破了 AI 与物理环境的隔阂。通过 MCP，AI 不再仅仅是一个对话机器人，它可以读取您的本地文件、执行代码、搜索网络、操作数据库，甚至控制其他软件，或是调度其他 AI 协同工作。

本文档将指导您如何配置和使用 MCP 服务，将您的 AI 升级为真正的“智能代理”。

---

## 1. 前置准备 (必读)

在使用 MCP 功能之前，请务必确保您的运行环境满足以下要求，否则服务将无法启动。

### 1.1 环境安装

绝大多数外部 MCP 服务依赖于 Python 或 Node.js 环境运行。（内置工具除Python工具外均不依赖外部环境）

* **Python 环境**：如果您计划使用 Python 相关的工具（如内置的 Python 执行器），请前往 [Python 官网](https://www.python.org/) 或 [Anaconda官网](https://anaconda.org/)下载、安装。
* **Node.js 环境**：如果您使用的是基于 `npx` 或 JavaScript 编写的 MCP 服务，请前往 [Node.js 官网](https://nodejs.org/) 下载并安装。
* **Git**：部分工具可能需要 Git 支持。

**⚠️ 重要提示：**
安装完上述环境后，**必须完全退出并重启 uTools**，或者**重启您的电脑**，以确保 Anywhere 能够读取到最新的系统环境变量（PATH）。

### 1.2 模型要求

MCP 的核心机制是 **Function Calling (工具调用)**。

* **推荐模型**：Claude 3.5 Sonnet, GPT-4o, DeepSeek-Chat(V3) 等官方旗舰模型，这些模型对工具调用的指令遵循度极高。
* **不推荐模型**：早期模型（能力较弱），针对 Function Calling 优化的模型。它们可能无法正确触发工具或无法解析工具返回的结果。

## 2. 内置 MCP 服务 (开箱即用)

Anywhere 为了让您开箱即用，预置了一组高度优化的内置服务。建议您在“MCP 服务”页面中保持它们的**启用**状态，它们是许多进阶功能（如 Skill 技能库和定时任务）的基础。

* **🤖 Super-Agent (超级智能体调度中心)**：
  * 这是核心的编排模块，赋予了当前 AI **多 Agent 协作**的能力。
  * **后台模式 (Sub-Agent)**：可生成隐形的后台 Worker 执行拆分步骤，专注完成单一任务并只返回最终结果，避免污染主对话上下文。
  * **前台调度**：AI 能够检索当前活跃的其他独立窗口 Agent、读取其聊天记录，甚至可以**新开一个特定角色**的独立窗口并在后台执行，实现 AI 之间的串联工作流。
* **📂 File Operations (全能文件操作)**：
  * 支持文件的读取、创建、编辑。
  * 特别强化了对代码文件的处理能力：支持 `Glob` 模式批量匹配文件路径，支持 `Grep` 正则搜索内容并提供上下文，支持精确的字符串替换及正则行号插入。
* **🐍 Python Executor**：
  * 自动扫描并检测本地的 Python 环境 (包括 Conda)。
  * AI 可动态编写并运行分析脚本，获取终端输出结果。
* **💻 Shell Executor**：
  * 执行终端命令（Win下为 PowerShell，Mac/Linux 下为 Bash）。
  * **长时任务支持**：支持启动如 `npm run dev` 类的后台服务，并提供读取日志和销毁进程的工具。
* **🌐 Web Toolkit**：
  * 包含使用 DuckDuckGo 免费搜索以及抓取指定 URL 的长文内容解析工具。
* **⏰ Task Manager**：
  * 定时任务的元管理服务。AI 可借此检索、新建或修改系统级别的定时任务计划。
* **🕒 Time Service**：获取当前系统时间及指定时区的日期。

## 3. 添加自定义 MCP 服务

您可以接入社区成千上万的开源 MCP 服务。点击界面底部的 **“+ 添加 MCP 服务”**。

### 3.1 配置类型 (Type)

#### Stdio (标准输入输出)

这是最常见的本地 MCP 类型。Anywhere 会在后台启动一个子进程来运行服务。

* **命令 (Command)**：启动服务的程序，如 `npx`、`python`、`uv`、`docker`。
* **参数 (Args)**：传递给命令的参数，**每行一个**。
  * *示例 (顺序思维 MCP)*：
    * Command: `npx`
    * Args:
      ```text
      -y
      mcp-sequentialthinking-tools
      ```
* **环境变量 (Env)**：如果服务需要 API Key（如 Github MCP），在此处配置 `KEY: VALUE` 或者 `KEY=Value`。

![img](..\image\MCP配置界面.png)

#### SSE / HTTP

用于连接远程或本地以 HTTP 服务形式运行的 MCP 服务器。

* **URL**：输入服务的完整地址，如 `http://localhost:3000/sse`。

### 3.2 关键设置

* **持久连接 (Persistent Connection)**：
  * **关闭 (默认)**：即用即走。AI 决定使用工具时启动进程，使用完后销毁，最大程度节省电脑内存。
  * **开启 (ON)**：服务启动后常驻后台，不会断开。
  * *适用场景*：需要保持状态的服务（如数据库长连接、需长时间保留内存的 REPL 环境），或是启动非常缓慢的大型服务。
  * *注意*：由于 uTools 线程限制，同时存在的外部持久连接最多支持 **5个**（内置服务不受此限制）。

## 4. 在对话中使用 MCP

### 4.1 启用服务与工具

在对话窗口的输入框下方工具栏，点击 **🛠️ (MCP工具)** 图标，勾选您本次对话想要赋予 AI 的能力集合。

* **快捷指令**：在输入框输入 `@` 可以快速呼出面板筛选并切换 MCP 服务。
* **精细化控制**：在弹出的面板中展开某个 MCP 服务，您可以**关闭该服务下的特定工具**，防止 AI 误用。

### 4.2 AI 如何调用

启用后，您**无需**告诉 AI 具体的代码细节。直接用自然语言提出需求，AI 会自动判断并调用。

* *用户*：“帮我用 grep 找一下 src 目录下所有用到 handleSubmit 的文件。”
* *AI (后台)*：调用 `grep_search` 工具。
* *AI (回复)*：“找到了，在 App.vue 的第 42 行……”
* *用户*：“打开一个专门写文案的 Agent 帮我把刚才的总结润色一下。”
* *AI (后台)*：调用 `summon_agent` 新开一个文案助手的窗口，并通过 `read_agent_chats` 等待其结果。

### 4.3 审批机制 (安全锁)

为了防止 AI 执行危险操作（如删除核心文件、发送敏感数据），Anywhere 提供了完备的审批机制。

* **手动批准 (默认)**：当 AI 试图调用工具时，对话流中会浮现卡片，显示工具名和参数。此时进程挂起，需您点击“确认”后才会真正向系统发送指令。
* **自动批准**：在每个工具气泡或全局 MCP 弹窗中勾选 **“自动批准工具调用”**。开启后，AI 的工作流将如丝般顺滑，适合在绝对信任的环境或自动化场景下使用。

## 5. 常见问题 (FAQ)

**Q1: 添加了 Stdio 服务，但连接测试一直转圈或失败？**

> **A:**
>
> 1. 请检查 **前置准备** 中的环境是否安装，且 PATH 环境变量已生效（请重启电脑）。
> 2. 如果是 `npx` 命令，请确保您的网络可以访问 npm 镜像源。
> 3. 尝试使用绝对路径。例如在 Windows 上将 `python` 改为 `C:\Python39\python.exe`。

**Q2: 开启了“持久连接”，为什么提示连接数已满？**

> **A:** 为了保证 uTools 的稳定性，Anywhere 限制最多同时存在 **5个** 第三方持久化进程。请在 MCP 列表页面关闭一些不常用的服务的“持久连接”开关（改为按需加载）。内置服务由底层直接接管，不占用此配额。

**Q3: 为什么 AI 总是胡乱调用工具或者参数格式错误（如 JSON 解析失败）？**

> **A:** 这强烈依赖于 **AI 模型的指令遵循能力**。请务必使用尖端模型。能力较弱的模型可能无法理解用户意图，从而编造参数。

**Q4: 我修改了 MCP 的环境变量，需要重启插件吗？**

> **A:** 不需要。您可以点击 MCP 列表页面右上角的 **“强制刷新”** 按钮，或者在对话界面 MCP 面板再次勾选该服务，系统会自动断开旧连接并加载最新配置。

**Q5: MCP 和 Skill 有什么区别？**

> **A:**
>
> * **MCP** 是**工具 (Tools)**：是 AI 的“手”，提供了原子化的能力（如：读文件、执行终端命令）。
> * **Skill** 是**流程 (SOP)**：是 AI 的“技术规范”，是一套定义好的工作规范或知识库（如：代码审查原则、周报排版格式）。
> * *关系*：Skill 往往包含文字指导，要求 AI 去正确地调用 MCP 工具来完成特定工作。
